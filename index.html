Iniciando repositório do GIT

Repositório do GIT é o mesmo que a pasta do projeto, para iniciar o repositório basta abrir o cmd na pasta do projeto e usar o comando:

cd c:/pasta/do/projeto
git init

README.md

Arquivo explicando o objetivo do projeto, assim como observações e instruções gerais do projeto. Padronizado o nome README.md e sempre fica na raiz do projeto ou repositório

git status

Comando usado para consultar o status do monitoramento ou "track" do GIT. Aqui são mostrados os arquivos que estão sendo monitorados (em verde) e os que não estão (em vermelho)

git status

git add

Comando usado para adicionar arquivos ao monitoramento ou "track" do GIT
Use o comando abaixo para adicionar arquivos individualmente:

git add <nome_do_arquivo>

ou
Use o comando abaixo para adicionar todos os arquivos de uma única vez:

git add -A

git commit

Comando para "salvar" as alterações do arquivo em determinada versão do seu projeto.

Comando abaixo serve para executar a ação de commit

git commit -m "Breve resumo das alterações realizadas entre o último commit e esse atual"

Comando abaixo serve para executar a ação de commit, mais a adição dos arquivos que não estavam sendo monitorados. É o mesmo que usar o "git add -A" e depois o "git commit -m 'descrição'"

git commit -am "Breve resumo das alterações realizadas entre o último commit e esse atual"

branchs

Branchs são como versões "braços"/"galhos" do projeto... branchs diferentes podem ser criados para determinada versão do projeto, ou para auxiliar no processo de diferentes equipes atuando em diferentes etapas do projeto. Ex: uma equipe trabalha no branch de produção do projeto, ou seja, na corrigindo eventuais falhas na versão que está em uso atualmente. Enquanto outra equipe atual num branch de melhorias para a versão futura do sistema.

Comando para ver quais branchs existem no repositório

git branch

Comando para criar um novo branch

git branch nome_do_branch

checkout

Comando usado para alternar entre branchs.

Exemplo abaixo leva o projeto para o branch 1.1:

git branch 1.1

Já esse, eva o projeto para o branch master:

git branch master

git log

Comando usado para visualizar o log de todos os commits realizados naquele branch

git log

reset

Comando usado para retornar o branch à determinado ponto (commit)

Há algumas opções diferentes de reset:

--soft: é realizado o reset no commit, mas as alterações dos arquivos e os commits ainda estarão disponíveis para que você consiga recupar essas alterações se necessários

--mixed: é realizado o reset no commit, as alterações dos arquivos são mantidas, mas o commit não, ou seja, os arqivos deverão ser adicionados novamente para depois ser realizad o commit

--hard: é realizado o reset no commit e nos arquivos, as alterações são completamente descartadas, sendo perdido qualquer informação dessas alterações.


Para usar o comando, é necessário ter o id/hash do commit, esse id é obtido com o "git log", pode ser usado o comando com os 7 primeiros dígitos do hash, ou com o hash inteiro. Abaixo o exemplo do comando:

git reset --soft fbc701a064b7e393905db5881cfd4549b93b537d git reset --hard fbc701a

diff

Comando usado para visualizar a diferença entre o último commit e o estado atual do projeto. O commando mostra arquivo por arquivo e linha por linha o que foi excluído e o que foi criado no projeto.

git diff

Para exibir somente o nome dos arquivos que foram alterados

git diff --name-only

Para exibir somente as alterações de um unico arquivo

git diff nome_do_arquivo

Para restaurar/excluir as alterações de um único arquivo. Exemplo: você verificou as diferenças de todos os arquivos com o "git diff" e descobriu que somente um arquivo deve ser ignorado, ou seja, somente um arquivo deve permanecer sem as alterações os demais serão commitados normalmente:

git checkout HEAD -- nome_do_arquivo_que_sera_restaurado

git remote

Comando usado para vincular o repositório local com um remoto

git remote add nome_repositorio link_do_repositorio_remoto
git remote add origin https://github.com/trindadekaio/treinamento_pzap-git

Nesse caso é necessário ter uma SSH Key configurada no computador. Ver os links de passo-a-passo disponíveis no painel do curso

O "nome_repositorio" do comando, é o nome de um branch que será criado no repositório local. Esse branch será usado quando for realizado o processo de enviar o repositório para o remoto (push) ou o processo de puxar o repositório remoto para o local (fetch). Normalmente esse nome é "origin" mas pode ser definido como quiser



Para verificar os repositórios remotos configurados

git remote

Para ver o endereço e ações disponíveis para cada repositório remoto

git remote -v

git push

Comando usado para enviar o repositório local para o remoto


Necessário realizar o add e commit antes de realizar esse processo.

git push -u origin master

Sempre considerar o repositório destino (origin) e depois o origem (master ou qualquer outro repositório que será usado)

git fetch

Comando usado para puxar o repositório remoto para o local


Necessário realizar o vinculo do repositorio remoto antes, ver o comando "git remote"

git fetch nome_repositorio

Esse "nome_repositorio" é o mesmo que foi definido quando foi criado o vínculo via "sgit remote"

git pull

Comando usado para puxar o repositório remoto para o local


Necessário realizar o vinculo do repositorio remoto antes, ver o comando "git remote"

git pull nome_repositorio

Esse "nome_repositorio" é o mesmo que foi definido quando foi criado o vínculo via "sgit remote"

.gitigore

Arquivo que deve ficar na raiz do repositório contendo os arquivos ou tipos de arquivos que devem ser ignorados pelo Git. Cada linha do arquivo, contem o nome do arquivo ou algum padrão de arquivo a ser ignorado. Exemplo de conteudo do arquivo:

*.sql *.txt

git revert

Comando usado para reverter um commit, diferente do reset, esse comando desfaz as alterações, ou seja, volta pra determinado commit, mantendo aquele commit no log. Com isso é possível voltar ao ponto revertido para analisar algum problema ou algo do tipo.
Exemplo: ao se deparar com um erro de uma versão nova do sistema, é possível realizar um revert, mantendo todas as alterações no commit da atualização, com isso o sistema volta ao ar, de forma ágil, mas ainda é possível verificar o erro com calma.

git revert --no-edit hash_id_do_commit
